# 各平台脚本使用说明

本文档说明 `FlowPilot` 守护脚本在 Windows、macOS、Linux 三个平台的使用方式。

## 1. 脚本清单

- Windows: `scripts/flow-daemon.ps1`
- macOS: `scripts/flow-daemon-macos.sh`
- Linux: `scripts/flow-daemon-linux.sh`

## 2. 功能概览

守护脚本会循环执行以下逻辑：

- 检查工作流状态文件 `.workflow/progress.md`。
- 如果没有活跃工作流，进入等待，不会拉起 `codex`。
- 若存在活跃工作流，先执行 `node <flow-cli> resume`。
- 检测到 `reconciling` 时暂停自动续跑，等待人工 `adopt / restart / skip`。
- 非 `reconciling` 才拉起 `codex exec resume --last`。
- 进程异常退出或日志命中上下文异常关键字时，按指数退避自动重启。

## 3. 前置要求

- 已安装 `Node.js`，命令 `node` 可用。
- 已安装 `codex`，命令 `codex` 可用。
- 项目根目录存在以下任一 CLI 入口：
  - `flow.js`
  - `dist/flow.js`

## 4. Windows 使用（PowerShell）

### 4.1 启动

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\flow-daemon.ps1 -ProjectRoot "E:\progamecode\AIprogram\test\autoAi\FlowPilot"
```

### 4.2 常用参数

- `-ProjectRoot`：项目根目录，默认当前目录。
- `-PollSec`：轮询间隔秒数，默认 `8`。
- `-MaxRestart`：最大重启次数，默认 `999`。
- `-MaxBackoffSec`：指数退避上限秒数，默认 `60`。
- `-ResumePrompt`：给 `codex exec resume --last` 的提示词。
- `-NoDangerousBypass`：加上后不使用 `--dangerously-bypass-approvals-and-sandbox`。

示例（降低日志频率）：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\flow-daemon.ps1 -ProjectRoot "E:\progamecode\AIprogram\test\autoAi\FlowPilot" -PollSec 30
```

## 5. macOS 使用（bash）

### 5.1 启动

```bash
chmod +x ./scripts/flow-daemon-macos.sh
./scripts/flow-daemon-macos.sh /path/to/FlowPilot
```

### 5.2 环境变量

- `POLL_SEC`：轮询间隔秒数，默认 `8`。
- `MAX_RESTART`：最大重启次数，默认 `999`。
- `MAX_BACKOFF_SEC`：指数退避上限秒数，默认 `60`。
- `RESUME_PROMPT`：传给 `codex` 的提示词。
- `NO_DANGEROUS_BYPASS=1`：关闭危险绕过参数。

示例：

```bash
POLL_SEC=20 NO_DANGEROUS_BYPASS=1 ./scripts/flow-daemon-macos.sh /path/to/FlowPilot
```

## 6. Linux 使用（bash）

### 6.1 启动

```bash
chmod +x ./scripts/flow-daemon-linux.sh
./scripts/flow-daemon-linux.sh /path/to/FlowPilot
```

### 6.2 环境变量

与 macOS 一致：

- `POLL_SEC`
- `MAX_RESTART`
- `MAX_BACKOFF_SEC`
- `RESUME_PROMPT`
- `NO_DANGEROUS_BYPASS`

## 7. 停止守护

推荐使用 `STOP` 文件方式优雅停机：

- 路径：`.flowpilot/daemon/STOP`

示例：

```powershell
New-Item -ItemType File -Path .\.flowpilot\daemon\STOP -Force
```

```bash
mkdir -p ./.flowpilot/daemon
touch ./.flowpilot/daemon/STOP
```

> 守护进程检测到 `STOP` 文件后会自行退出，并清理 `.flowpilot/daemon/daemon.pid`。

## 8. 日志与状态文件

- `daemon pid`: `.flowpilot/daemon/daemon.pid`
- `stop flag`: `.flowpilot/daemon/STOP`
- `codex run log`: `.flowpilot/daemon/codex-YYYYMMDD-HHMMSS.log`

## 9. 典型输出含义

- `[daemon] no workflow state file yet, waiting.`
  - 说明尚未初始化工作流或 `.workflow/progress.md` 不存在。
- `[daemon] no active workflow, waiting.`
  - 说明当前不是运行中的工作流（如 `idle/completed/aborted`）。
- `reconciling detected; auto-resume paused...`
  - 说明命中安全边界，必须人工处理，不会自动 `adopt`。
- `restart #N after X sec.`
  - 说明触发自动重启，退避等待后重试。

## 10. 推荐操作顺序

1. 先启动守护脚本。
2. 在另一个终端执行 `node dist/flow.js init`（或 `node flow.js init`）初始化工作流。
3. 守护脚本检测到活跃工作流后开始自动续跑。
4. 若进入 `reconciling`，按提示人工执行：
   - `node <flow-cli> adopt <id> --files ...`
   - 或 `node <flow-cli> restart <id>`
   - 或 `node <flow-cli> skip <id>`

## 11. 常见问题

### Q1: 一直打印 `no workflow state file yet, waiting.`

A1: 这是正常待机状态。请确认你已在同一项目目录执行过 `init`，且生成了 `.workflow/progress.md`。

### Q2: 报错找不到 CLI

A2: 需要保证项目根目录存在 `flow.js` 或 `dist/flow.js`。如果你是源码仓库，通常使用 `dist/flow.js`。

### Q3: 为什么不自动 `adopt`

A3: `adopt` 是“变更归属认领”动作，涉及用户手工改动风险。脚本设计为在 `reconciling` 阶段暂停并等待人工确认，防止误认领。
