# FlowPilot 规则新增白名单

这份文档用于约束 FlowPilot instruction template 的新增规则应该放在哪一层，避免把风格要求、调度策略和协议体硬规则混在一起，进而破坏工作流闭环。

## 一句话判断法

- “怎么说” → 风格层
- “怎么派活” → 调度策略层
- “什么时候跑哪个 `flow.js` 命令” → 协议体硬规则层

## 三层边界

### 1. 风格层

只影响表达方式，不影响调度顺序、状态机和命令闭环。

适合新增的内容：

- 语言要求：简体中文、双语切换规则
- 输出结构：先结论后细节、标题分组、留白
- 展示规范：代码块、日志块、ASCII 图示、避免超长表格
- 终端可读性：控制单行长度、避免超长路径、突出重点
- 语气约束：简洁、直接、少废话、避免过度客套

推荐放置位置：

- `COMMON_AGENT_GUIDELINES`

### 2. 调度策略层

影响任务拆解、并行派发、汇总与递归迭代，但不直接改 FlowPilot 生命周期命令。

适合新增的内容：

- 任务拆解方法：依赖关系图、先拆边界再并行、先收集再分析
- 并行策略：无前置依赖优先并发、有写冲突先拆分、超并发后分批
- 子任务规范：代理名称、任务定义、执行动作、预期结果、输出格式
- 汇总策略：等待全部返回、冲突检查、阶段性结果收敛、递归下一轮
- 安全策略：高风险操作确认、数据操作确认、生产环境操作确认
- 质量策略：测试优先、验证优先、可追溯、证据优先

推荐放置位置：

- `CODEX_ENHANCED_GUIDELINES`
- 通用能力足够稳定时，再考虑下沉到 `COMMON_AGENT_GUIDELINES`

### 3. 协议体硬规则层

直接决定 FlowPilot 的工作流状态机和命令闭环。除非明确要修改产品协议，否则不要动。

属于这一层的内容：

- `On Session Start`
- `Ad-hoc Dispatch`
- `Requirement Decomposition`
- `Execution Loop`
- `Sub-Agent Prompt Template`
- `Sub-Agent Checkpoint`
- `Finalization`
- `resume / init / next / checkpoint / review / finish`
- `reconciling / adopt / restart / skip`
- `finish -> review -> finish` 收尾闭环

对应位置：

- `FLOWPILOT_PROTOCOL_BODY`

## 默认白名单

### 可直接新增到风格层

- 简体中文沟通
- 先结论后细节
- 强视觉边界
- 短段落、短列表、代码块、ASCII 图示
- 终端友好排版
- 控制冗长表格和大段无结构文本

### 可新增到调度策略层

- 最大化并行、最小化阻塞
- 依赖关系图
- 无前置依赖优先并发
- 写冲突先拆边界
- 结果汇总后递归下一轮
- 子任务契约四要素
- 并发上限与分批策略
- 危险操作确认模板
- 质量与验证要求

## 默认不要加的内容

以下内容高风险，默认不要直接新增：

- 改命令顺序
- 给协议体新增另一套更高优先级流程
- 改 `checkpoint --files` 含义
- 改 `resume` 的恢复条件
- 改 `next --batch` 的前置条件
- 改 `finish` / `review` 的闸门关系
- 把建议写成覆盖硬规则的“必须”
- 把某个客户端特有规则强塞给所有客户端
- 把模板写成超长手册，稀释真正高优先级协议

## 新增规则判断流程

1. 先判断这条规则是在管表达、调度，还是在管状态机。
2. 如果只是表达方式，放到风格层。
3. 如果是拆任务、派子代理、做汇总，优先放到调度策略层。
4. 如果会影响 `resume/init/next/checkpoint/finish`，按高风险协议改动处理。
5. 任何模板改动完成后，至少执行一次 `npm run build` 验证产物生成。

## 当前实践

目前 FlowPilot 已经把以下内容放在风格层或调度策略层，而没有写进协议体硬规则层：

- 输出风格硬约束
- 危险操作确认模板
- 高风险操作清单
- 质量标准
- `Codex` 标准执行流程
- `Codex` 子任务契约四要素

这样做的目的很直接：最大化吸收高价值规范，同时不破坏 `resume/init/next/checkpoint/finish` 这套真正决定工作流闭环的协议体。
